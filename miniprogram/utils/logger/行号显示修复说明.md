# Logger 行号显示功能修复说明

## 问题描述

之前的日志系统虽然实现了调用栈追踪功能，但在实际使用中只显示了类名，没有显示方法名和行号：

```
❌ 旧输出：
[2025-10-12 15:45:15.417] [INFO] [card] [CardController] 等待ProfileManager初始化完成...
```

## 问题原因

在 `BaseClass.js` 的日志方法中，传递了 `this.className` 作为 `caller` 参数：

```javascript
// 问题代码
logger.info(this._logModule, message, data, this.className);
```

而在 `Logger.js` 的 `_log()` 方法中，当 `caller` 参数有值时，会直接使用它，不会调用 `getCallerInfo()` 来获取完整的调用栈信息：

```javascript
// Logger.js 第126行
const callerInfo = caller || this.getCallerInfo();
```

这导致日志只显示了类名，而没有方法名和行号信息。

## 解决方案

### 1. 修改 BaseClass.js

移除所有日志方法中的 `caller` 参数传递，让 Logger 自动获取完整的调用栈信息：

```javascript
// 修改前
_log(message, data = undefined) {
  logger.info(this._logModule, message, data, this.className);
}

// 修改后
_log(message, data = undefined) {
  // 不传递 caller 参数，让 Logger 自动获取完整的调用栈信息（包括行号）
  logger.info(this._logModule, message, data);
}
```

涉及的方法：
- `_log()`
- `_info()`
- `_warn()`
- `_error()`
- `_debug()`

### 2. 修改 Logger.js

更新 `getCallerInfo()` 方法，让它能够正确跳过框架层级的调用：

```javascript
// 修改前
for (let i = 4; i < lines.length; i++) {
  const line = lines[i];
  
  // 跳过 logger 内部调用
  if (line.includes('Logger.js') || line.includes('logger/')) {
    continue;
  }
  // ...
}

// 修改后
for (let i = 2; i < lines.length; i++) {
  const line = lines[i];
  
  // 跳过 Logger 内部调用和所有 Base 类调用（框架层级）
  if (line.includes('Logger.js') || 
      line.includes('logger/') || 
      line.includes('BaseClass.js') ||
      line.includes('BaseService.js') ||
      line.includes('BaseController.js') ||
      line.includes('BaseBean.js')) {
    continue;
  }
  // ...
}
```

**关键改进：**
- 从索引 2 开始遍历（而不是 4），确保不会跳过太多层级
- 跳过所有框架层级：Logger、BaseClass、BaseService、BaseController、BaseBean
- 返回最顶层的业务代码调用位置（如 UserBean、CardController 等）
- 这样可以定位到真正发起调用的业务代码位置

## 调用栈追踪机制

### 调用链路示例 1：Controller 中直接调用日志

```
CardController.initialize:234  ← 最顶层业务代码
    ↓
BaseClass._info()  ← 框架层（跳过）
    ↓
Logger.info()  ← Logger 内部（跳过）
    ↓
Logger._log()  ← Logger 内部（跳过）
    ↓
Logger.getCallerInfo()
    ↓
解析调用栈，返回 "CardController:initialize:234"  ✅
```

### 调用链路示例 2：基类中调用日志

```
UserBean.constructor:40  ← 最顶层业务代码（这是我们要的）
    ↓
BaseBean._normalizeData:65  ← 框架层（跳过）
    ↓
BaseClass._warn()  ← 框架层（跳过）
    ↓
Logger.warn()  ← Logger 内部（跳过）
    ↓
Logger._log()  ← Logger 内部（跳过）
    ↓
Logger.getCallerInfo()
    ↓
解析调用栈，返回 "UserBean:constructor:40"  ✅
```

**重要说明**：返回的是**最顶层的业务代码调用位置**，跳过所有框架层级（BaseClass、BaseService、BaseController、BaseBean）。这样可以清楚地知道是哪个业务类发起的调用。

### 跳过的框架层级

为了获取真实的业务调用位置，Logger 会自动跳过以下框架层级：

1. **Logger.js 和 logger/ 目录** - Logger 内部调用
2. **BaseClass.js** - 基类的所有方法（包括日志方法和其他方法）
3. **BaseService.js** - Service 基类的所有方法
4. **BaseController.js** - Controller 基类的所有方法
5. **BaseBean.js** - Bean 基类的所有方法

**目标**：返回最顶层的业务代码位置（如 UserBean、CardController、ProfileService 等具体的业务类）。

## 支持的调用栈格式

Logger 的 `parseStackLine()` 方法支持多种调用栈格式：

1. **模式1：类方法调用**
   ```
   at ClassName.methodName (file:line:col)
   → 输出：ClassName:methodName:line
   ```

2. **模式2：对象方法调用**
   ```
   at Object.methodName (file:line:col)
   → 输出：Object:methodName:line
   ```

3. **模式3：函数调用**
   ```
   at methodName (file:line:col)
   → 输出：methodName:line
   ```

4. **模式4：直接文件调用**
   ```
   at file:line:col
   → 输出：file:line
   ```

## 使用建议

### 1. 继承自 BaseClass 的类

直接使用 `this._log()`、`this._info()` 等方法即可，自动获取行号：

```javascript
class CardController extends BaseController {
  async loadCard() {
    this._info('开始加载卡牌');  // 自动显示 CardController:loadCard:123
    // ...
  }
}
```

### 2. 非类代码

使用 `createModuleLogger` 创建模块日志器：

```javascript
const { createModuleLogger } = require('@logger/');
const log = createModuleLogger('UserManager');

log.info('用户登录成功', { userId: 123 });  // 自动显示 filename:line
```

### 3. 直接使用 Logger

```javascript
const logger = require('../utils/logger/Logger');

logger.info('user', '操作成功');  // 自动显示 调用位置:line
```

## 性能影响

- **开销**：每次日志调用需要解析调用栈（约 0.1-0.5ms）
- **优化**：生产模式只记录 WARN 和 ERROR，大幅减少开销
- **建议**：在性能敏感的循环中，避免频繁调用 DEBUG/INFO 日志

## 测试示例

### 场景 1：业务类中直接调用日志

```javascript
// CardController.js
class CardController extends BaseController {
  async initialize() {
    this._info('开始初始化卡牌页面');  // 第234行
    // ...
  }
}
```

**输出：**
```
[INFO] [card] [CardController:initialize:234] 开始初始化卡牌页面
```

### 场景 2：通过基类间接调用日志

```javascript
// UserBean.js（第40行）
class UserBean extends BaseBean {
  constructor(data) {
    super(data);  // 内部会调用 BaseBean._normalizeData()，而它又会调用 this._warn()
    // ...
  }
}

// BaseBean.js（第65行）
class BaseBean extends BaseClass {
  _normalizeData(data) {
    if (!data || typeof data !== 'object') {
      this._warn('数据不是有效的对象，使用空对象', { data });
      return {};
    }
    // ...
  }
}
```

**输出：**显示最顶层的业务代码位置（UserBean），而不是中间层的 BaseBean
```
[WARN] [bean] [UserBean:constructor:40] 数据不是有效的对象，使用空对象
```

### 场景 3：Service 中调用日志

```javascript
// ProfileService.js
class ProfileService extends BaseService {
  async getProfileList() {
    this._info('开始获取档案列表');  // 第156行
    // ...
  }
}
```

**输出：**
```
[INFO] [profile] [ProfileService:getProfileList:156] 开始获取档案列表
```

### 场景 4：普通函数中调用日志

```javascript
// utils/someHelper.js
const { createModuleLogger } = require('@logger/');
const log = createModuleLogger('helper');

function processData(data) {
  log.info('开始处理数据', { count: data.length });  // 第45行
  // ...
}
```

**输出：**
```
[INFO] [helper] [processData:45] 开始处理数据
```
